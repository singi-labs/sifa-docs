Sifa's primary and secondary brand colors come from the [Flexoki](https://stephango.com/flexoki) palette, exposed as constants via the `/tokens` subpath of [`@singi-labs/sifa-sdk`](https://www.npmjs.com/package/@singi-labs/sifa-sdk) (currently <SdkVersion inline />).

## Live values

<SdkColors />

## Type

```ts
interface Colors {
  /** Flexoki Blue: Sifa's primary accent. */
  primary: string
  /** Flexoki Purple: secondary accent, shared across Singi Labs products. */
  secondary: string
}
```

Exported as a `const` object at runtime, so TypeScript narrows the values to their literal string types. Updates to the palette propagate via `pnpm update @singi-labs/sifa-sdk` without find-and-replace.

If you're styling with CSS, reference the same hex values as variables. Sifa Web and Sifa Docs both expose them as CSS variables (`--brand-blue-solid`, etc.); see [`sifa-web/src/app/globals.css`](https://github.com/singi-labs/sifa-web/blob/main/src/app/globals.css) for the variable names.

## Why not just the hex values?

Hex values change rarely, but they do change. Importing from `@singi-labs/sifa-sdk/tokens` means a future Sifa-wide rebrand or a Flexoki update propagates to your app via `pnpm update @singi-labs/sifa-sdk`, no find-and-replace required. It also keeps every Sifa-aware app rendering the same color, which matters if you build a third-party Sifa client meant to feel native.

## Where this fits

Sifa's full design system (typography, spacing, motion) is in [Singi Labs / brand](https://github.com/singi-labs) and not part of the SDK. The `/tokens` subpath only exposes the cross-platform primitives a third-party client needs to render a Sifa-flavoured UI without depending on Tailwind or a specific framework.
